# 乔布斯视角：五层爆款图谱的产品设计审查

> **日期**: 2026-06-16
> **审查维度**: 优先级、为什么是5层、数据来源、评论回流
> **方法**: 乔布斯式产品哲学

---

## 一、优先级：哪一层最重要？

### 乔布斯会说：中心即一切。

> "People think focus means saying yes to the thing you've got to focus on. But that's not what it means at all. It means saying no to the hundred other good ideas."

如果五层架构是 iPod——

- ⑤ 信号环 = 包装盒背面的规格表。没人因为规格表买 iPod。
- ④ 证据环 = 广告里说"1000 songs in your pocket"。它说服你，但它不是产品。
- ③ 候选基因环 = 滚轮旁边的按钮。有用，但滚轮才是革命。
- ② 约束环 = 电池。没电不行，但用户不关心电池长什么样。
- **① 场域中心 = 滚轮。** 这是你触摸的第一个东西。这是为什么这个产品存在的唯一理由。

### 优先级排序（从用户体验出发）

| 优先级 | 层 | 乔布斯式理由 |
| --- | --- | --- |
| **P0** | ① 场域中心 | "如果用户打开这个工具，30 秒内看不懂问题在哪，她永远不会用第二次。" |
| **P1** | ② 约束环 | "约束是自然法则。你不能和物理辩论。这一层必须绝对正确，否则后面的推理都是垃圾。" |
| **P2** | ④ 证据环 | "人们信看到的东西。如果这一层没有让设计师产生'哦，原来是这样'的瞬间——删掉重做。" |
| **P3** | ③ 候选基因环 | "这是推理引擎。它重要，但它是①和②的产物。如果①②错了，③只是一堆好看的假数字。" |
| **P4** | ⑤ 信号环 | "这是给老板看的。设计师不需要这一层来做好设计。但它让决策者掏钱。" |

### 用户阅读顺序 ≠ 逻辑依赖顺序

```
逻辑依赖：① → ② → ③ → ④ → ⑤（内向外，一层推导一层）
用户阅读：① → ② → ④ → ③ → ⑤（看到问题→看到规则→看到证据→看到选项→看到决策）
```

乔布斯会坚持**用户阅读顺序**作为设计顺序。因为"I want to see the evidence that this works"在用户心智中排在"I want to see the engine that generated the options"之前。

## 二、为什么是 5 层？不是 3 层，不是 7 层？

### 乔布斯式试错

**3 层会怎样？**
```
① 场域 → ② 约束 → ⑤ 信号
```
你告诉企划师"这里有需求空隙，这个空隙长这样，我们的满足度 6.2"——然后呢？她不知道**怎么做**。不知道哪些属性组合被验证过。她只能回到旧习惯。**3 层不完整。**

**4 层会怎样？**
```
① 场域 → ② 约束 → ③ 候选 → ⑤ 信号
```
你告诉企划师"约束翻译成了这些候选方案"——但她敢信吗？"系统说天丝混纺得分 91，但万一只是算法自以为呢？"**缺了证据，就没有信任。**

**6 层会怎样？**
在候选和证据之间插入"流行趋势环"？在信号之外再加"财务预测环"？**每加一层，阅读路径多一个弯，30 秒变 3 分钟，没人用了。**

**5 层 = 刚好。**
```
① 问题在哪 → ② 什么是必须 → ③ 什么可能 → ④ 什么已验证 → ⑤ 该不该打
```
五个问题，五种信息类型，五种不可合并的认知功能。**砍任何一层，链条断裂。加任何一层，噪音超过信号。**

### 每一层的不可替代性

| 如果砍掉这一层 | 用户会失去什么 |
| --- | --- |
| 砍① | 盲人。不知道看哪里。 |
| 砍② | 无根。候选方案变成"算法黑箱推荐"，像淘宝"猜你喜欢"——设计师不会信。 |
| 砍③ | 断桥。约束有了，证据有了，中间怎么推导的不知道。不可解释。 |
| 砍④ | 无信。全是算法推理，没有任何市场事实。没人敢下单。 |
| 砍⑤ | 无力。数据都有了，但没人把它翻译成"你现在该不该做"。 |

## 三、数据从哪来？是合成数据吗？

### 不是合成数据。但也不是 100% 真实数据。

是**三层数据架构**：

```
┌─────────────────────────────────────────┐
│  第三层：推理数据（合成）                │
│  ③ 候选基因环                           │
│  "约束 X+Y+Z → 哪些属性同时满足？"       │
│  来源：知识图谱推理引擎                   │
│  本质：逻辑推导，不是观测                 │
├─────────────────────────────────────────┤
│  第二层：计算数据（派生）                │
│  ① 场域的满足度分数                       │
│  ⑤ 优先级评分                            │
│  ④ 基因覆盖匹配度                        │
│  来源：真实数据 + 算法加工                │
│  本质：真实数据的结构化归纳               │
├─────────────────────────────────────────┤
│  第一层：真实数据（观测）                 │
│  ② 物理约束 ← 气象数据库                 │
│  ② 身体约束 ← CRM 尺码/退换货原因        │
│  ④ 自有销售数据 ← ERP                   │
│  ④ 竞品情报 ← 第三方/爬虫/人工录入       │
│  ⑤ 客诉/复购/流失 ← CRM                 │
│  本质：不可捏造的观测事实                 │
└─────────────────────────────────────────┘
```

### 逐层数据溯源

| 层 | 数据项 | 来源 | 合成？ |
| --- | --- | --- | --- |
| ① | 场域定义（觅寻线/行走线） | 品牌战略部人工定义 | **否——人定** |
| ① | 活跃 VIP 数 | CRM 数据库 | **否——实时** |
| ① | 满足度分数 | ⑤ 的计算结果回灌 | **派生** |
| ② | 温度/湿度约束 | 中国气象局公开数据 + 商品上市日历 | **否——真实** |
| ② | 社会约束（正式度等） | 设计师/造型师领域知识录入 + 客服评价 NLP 抽取 | **半合成——人机协作** |
| ② | 身体约束（遮/露/收） | CRM 尺码分布 + 退换货原因 NLP + 试衣间放弃行为 | **否——真实行为数据** |
| ③ | 约束→属性翻译 | 面料知识库 ∩ 版型知识库 ∩ 色域知识库 → 同时满足约束的所有属性组合 | **逻辑推理（可信合成）** |
| ③ | 候选得分 | 多约束同时满足度 × 历史相似场域表现 × 供应链可行性 | **算法评分（需可解释）** |
| ④ | 自有商品基因覆盖 | ERP 商品档案（面料/版型/颜色标签） × ③ 的候选基因 → 匹配度计算 | **匹配计算（真实档案×推理）** |
| ④ | 竞品基因覆盖 | 第三方竞品数据库 / 人工标注 / 公开信息 → 同款匹配算法 | **半合成（部分人工）** |
| ④ | VIP 截获率 | CRM 流失模型：近12月未复购 VIP 在竞品品牌的消费推断 | **算法推断（需标注不确定性）** |
| ⑤ | 场域满足度 | ② 的约束权重 × ④ 的基因覆盖 × VIP 复购行为 → 综合评分 | **派生** |
| ⑤ | 失分点 | ② 中高权重约束 ∩ ④ 中低覆盖基因 → 差集 | **派生** |
| ⑤ | 优先级 | 客群规模 × 满足度缺口 × 竞品验证 × 自有能力 → 排序 | **派生** |

### 乔布斯式判断

> "合成不可怕。可怕的是用户不知道什么是合成。"

- **③ 候选基因环**是合成数据——但它的每一步推导都来自 ② 的可追溯约束。点击任何一个候选基因，系统必须能回答："是因为 12-22°C 的约束 + 久坐不皱的约束 同时成立，才推荐天丝混纺。"
- **可解释性 = 信任。** 如果设计师问"为什么给我推这个"，系统回答不了——这一层就废了。

## 四、评论回流：数据怎么回来？

### 这是整个系统最重要的问题。没有之一。

五层图谱是一个**静态快照**。如果评论不回流，它只能用一次。第二季打开，数据是旧的——设计师用一次，失望，不再打开。

### 回流链路

```
设计师基于图谱决策
      ↓
  出图 → 打版 → 订货会 → 上架
      ↓
  顾客购买 / 试穿放弃 / 退货
      ↓
  ┌─────────────────────────────────┐
  │        评论与行为数据回流        │
  │                                 │
  │ ① 商品评论 NLP                  │
  │   "面料很舒服但坐久了有点皱"     │
  │   → ② 约束环更新：              │
  │   "久坐不皱"约束权重 +1          │
  │   天丝混纺得分 -3               │
  │                                 │
  │ ② 退换货原因 NLP                │
  │   "显胖" "颜色比图片暗"          │
  │   → ② 约束环：                 │
  │   "收而不紧"约束权重 +2          │
  │   → ③ 候选环：                 │
  │   冷灰调排除置信度提升           │
  │                                 │
  │ ③ 试衣间放弃行为               │
  │   试了但没买 →                  │
  │   与同场域购买商品做基因差集     │
  │   → ⑤ 信号环：失分点更新       │
  │                                 │
  │ ④ 复购 / 流失行为               │
  │   买了裤子没买外套               │
  │   第二季不再出现                 │
  │   → ① 场域中心：满足度重算      │
  │   → ⑤ 信号环：截获率更新       │
  │                                 │
  │ ⑤ 导购反馈（结构化录入）        │
  │   "顾客说想要能配裤子的外套"     │
  │   → ② 约束环：新增 "跨品类搭配" │
  │     社会约束                    │
  └─────────────────────────────────┘
      ↓
  场域满足度自动重算
      ↓
  证据环自动刷新：新商品家族基因交集更新
      ↓
  下一季设计师打开图谱——
  看到的是更新后的数据、新增的约束、更强的证据。
```

### 回流的不是"评论"，是**约束权重的更新**

乔布斯不会允许系统里出现一个叫"评论管理"的模块。那不是产品。那是功能。

回到产品里，评论应该变成什么？

> 评论不会以"评论"的形式出现在界面上。它会以**约束权重的变化**、**候选基因得分的升降**、**证据环新出现的验证卡片**的形式，**沉默地出现在下一季的图谱中**。

设计师看到的不是"有 376 条评论说这个面料皱了"。她看到的是"久坐不皱约束权重从 7 升到 9，天丝混纺候选得分从 91 降到 88"。

## 总结：乔布斯会怎么评价这个产品

| 问题 | 回答 |
| --- | --- |
| 优先级 | ① 场域中心是滚轮。其他的存在是为了支撑它。 |
| 为什么 5 层 | 5 个问题，5 种信息类型，砍任何一个链条断裂，加任何一个噪音超载。 |
| 数据是合成的吗 | ③ 是推理合成——可接受，但必须可追溯。其余全部根植于真实数据。 |
| 评论怎么回来 | 不显示评论。显示约束权重的更新、得分的升降、新证据的出现。 |

> **"Design is not just what it looks like and feels like. Design is how it works."**

这个产品的"how it works"不是五层架构本身。是——

> **一个设计师用了一次，做出来的衣服卖得比她以前好。第二季她主动打开，发现数据更新了，约束权重变了，新的竞品证据出现了——系统比上次更聪明了。这才是 how it works。**
