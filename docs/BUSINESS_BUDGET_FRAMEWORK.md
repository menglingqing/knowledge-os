## 一、核心公式

```
预算 = f(ARR, 服务量, 结构, 占比)
```

四个词不是独立指标，是一个推导链条。

---

## 二、四步推导

### 第一步：ARR 定盘子

```
明年 ARR 目标 = 今年 ARR × (1 + 增速%)

例：
  今年 ARR 5000 万，目标增速 40%
  明年 ARR 盘子 = 7000 万
```

ARR 决定后面所有数字的上限。**预算是 ARR 的函数。**

### 第二步：服务量 — 翻译层

ARR 是钱，但招人、买机器、租办公室按「量」算。服务量是中间翻译层。

```
ARR → 客户数 → 服务量

ARR 7000 万
  → 平均客单 7 万/年 → 1000 个客户
  → 1000 客户 × 3 次支持请求/月 → 36,000 次/年
  → 36,000 次 ÷ 1500 次/人/年 → 需要 24 个客服
```

**服务量是预算的脊椎。** 不对不叫预算，叫许愿。

### 第三步：结构 — 钱花在哪

```
总预算 = ARR × 预算率（通常 60-80%）
```

**结构 = 策略翻译成钱。** 重研发 → 产品驱动。重销售 → 市场驱动。

### 第四步：占比 — 比率体检

结构是绝对数，占比是比率。同一个数字，两个视角。

| 指标 | 公式 | 健康线 |
|------|------|--------|
| 人效 | ARR / 员工总数 | SaaS > $150K/人 |
| 研发占比 | 研发成本 / ARR | < 30% 健康，> 50% 未跑通 PMF |
| 获客效率 | CAC / ARPU | < 12 个月回本 |
| 毛利 | (ARR - 直接成本) / ARR | > 70% SaaS 健康 |
| 单位经济 | LTV / CAC | > 3 倍 |
| 预算偏差率 | 实际 / 预算 - 1 | > 20% 预测能力差 |

---

## 三、老板用这四个词找裂缝

| 信号 | 说明 |
|------|------|
| ARR 涨但服务量不涨 | 客单在涨，客户结构上移——好事 |
| ARR 涨服务量暴涨 | 客单在降，散户涌入，churn 风险 |
| 结构里研发占比降、销售占比升 | 产品转销售驱动，产品力可能不足 |
| 占比里人效在降 | 组织膨胀，收入没跟上 |
| NRR < 100% | ARR 靠新客冲量，老客在流失 |

---

## 四、AR 公司的预算地狱

```
ARR 7000 万
  → API 调用量：每天 5 亿 token
  → 推理成本：每百万 token $0.5 → 每天 $25 万 → 每年 $9125 万

  ARR 7000 万，推理成本 9125 万，还没算人。
  要么提价、要么降推理成本、要么接受战略性亏损。
```
